## 1. Executive Summary

This blueprint defines the complete technical architecture for the General Computing Solutions (GCS) website and client portal. The platform consists of two major components: a public-facing marketing website and a secure client portal with full project management, ticketing, and real-time communication capabilities.

The system is designed as a fully modular, component-based application with dark/light mode theming throughout, built to professional enterprise standards with comprehensive legal pages, error handling, and accessibility compliance.

**Key Deliverables:**

- Modern, responsive public website with full SEO optimization
- Client portal with authentication, project dashboards, and real-time updates
- Fully functional ticketing and support system with SLA tracking
- Real-time communication layer (chat, comments, file sharing)
- Email and in-app notification engine
- Project timeline with task tracking, milestones, and progress visualization
- Dark mode / Light mode with system preference detection
- Complete legal pages: Privacy Policy, Terms of Service, Cookie Policy
- Custom 404, 500, and maintenance pages

## 5. Public Pages Specification

### Home Page (/)

The landing page is the primary conversion driver. It must communicate GCS's value proposition within the first viewport and guide visitors toward either requesting a quote or exploring services.

1. **Hero Section:** Full-width gradient background (dark blue to primary blue), headline, subheadline, dual CTA buttons (Get a Quote, Explore Services), optional animated background element
2. **Trust Bar:** Client logos carousel, industry certifications, uptime stats
3. **Services Grid:** 4-6 service cards with icons, brief descriptions, hover effects, and links to detail pages
4. **Why GCS Section:** Three-column layout highlighting differentiators with animated counters (projects delivered, uptime %, years experience)
5. **Process Section:** Horizontal stepper showing Discovery → Design → Build → Deploy → Support
6. **Testimonials:** Rotating client testimonials with avatar, name, company, quote
7. **CTA Banner:** Full-width gradient with contact form or CTA button
8. **Footer:** Multi-column with navigation, contact info, social links, legal links, newsletter signup

### Services Pages (/services/*)

Each service page follows a consistent template structure:

- Hero with service-specific icon and overview
- Detailed feature breakdown with visual elements
- Use cases or industry applications
- Technology partners and tools used
- Related case studies
- Service-specific FAQ accordion
- CTA section with service-specific messaging

### Legal Pages

**Privacy Policy:** GDPR/CCPA compliant. Covers data collection, usage, cookies, third-party sharing, retention, user rights, and contact info.

**Terms of Service:** Covers service scope, user responsibilities, payment terms, IP ownership, liability limitations, termination, and dispute resolution.

**Cookie Policy:** Cookie categories, consent management, opt-out instructions, and third-party cookie disclosures.

All legal pages share a clean, readable layout with a sidebar table of contents for easy navigation, clear headings, last-updated dates, and contact information for legal inquiries.

### Error Pages

**404 Page:** Branded illustration, friendly message, search bar, popular page links, and home button. Maintains full site navigation.

**500 Page:** Minimal branded page (loads without server dependencies), apology message, auto-retry option, support contact, and status page link.

**Maintenance Page:** Static HTML file (no server needed), estimated downtime, status updates, social media links for updates.

## 7. Ticket & Support System

### Ticket Data Model

| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Unique ticket identifier |
| ticket_number | String | Human-readable ID (e.g., GCS-2024-0001) |
| subject | String | Ticket title / summary |
| description | Rich Text | Detailed description with markdown support |
| status | Enum | open, in_progress, waiting_client, waiting_gcs, resolved, closed |
| priority | Enum | low, medium, high, critical |
| category | Enum | bug, feature_request, support, billing, general |
| project_id | UUID (nullable) | Linked project if applicable |
| client_id | UUID | Submitting client/organization |
| assigned_to | UUID (nullable) | GCS team member assigned |
| created_at | DateTime | Submission timestamp |
| updated_at | DateTime | Last activity timestamp |
| resolved_at | DateTime (nullable) | Resolution timestamp |
| sla_deadline | DateTime | SLA response deadline based on priority |
| attachments | File[] | Uploaded screenshots, documents, files |
| tags | String[] | Searchable labels for categorization |

### Ticket Workflow

1. Client creates ticket via form (subject, description, category, priority, attachments)
2. System auto-assigns ticket number, sets SLA deadline, sends confirmation email
3. GCS admin receives notification, assigns to team member
4. Status transitions are logged with timestamps in the activity timeline
5. Both client and GCS team can add replies in a threaded conversation format
6. Each reply triggers email notification to the other party
7. Resolution triggers satisfaction survey prompt
8. Closed tickets are searchable in history with full conversation preserved

### Ticket Views

- **List View:** Sortable/filterable table with status badges, priority icons, and relative timestamps
- **Detail View:** Full ticket with threaded conversation, activity timeline, file attachments, and metadata sidebar
- **Kanban View:** Drag-and-drop board grouped by status columns (admin only)

## 8. Project Management & Timeline System

### Project Data Model

| Field | Type | Description |
|-------|------|-------------|
| id | UUID | Unique project identifier |
| name | String | Project display name |
| description | Rich Text | Project overview and scope |
| status | Enum | discovery, planning, in_progress, review, completed, on_hold |
| progress | Integer (0-100) | Overall completion percentage |
| start_date | Date | Project kickoff date |
| target_date | Date | Estimated completion date |
| client_id | UUID | Client organization |
| team_members | User[] | GCS team assigned |
| milestones | Milestone[] | Key project milestones |
| tasks | Task[] | Individual work items |
| requirements | Requirement[] | Client requirements/features |
| files | File[] | Project documents and deliverables |
| budget | Decimal (nullable) | Project budget if applicable |

### Timeline & Progress Features

- **Interactive Gantt Chart:** Visual timeline with milestones, task dependencies, and current date indicator. Draggable for zoom/scroll.
- **Progress Dashboard:** Overall percentage with breakdown by phase (Discovery, Design, Development, Testing, Deployment)
- **Milestone Tracker:** Key deliverable dates with completed/upcoming/overdue indicators
- **Task Board:** List or Kanban view of all tasks with status, assignee, and due date
- **Activity Feed:** Chronological stream of all project events (status changes, comments, file uploads, task completions)
- **Requirements Log:** Client can submit new feature requests or requirement changes, which are reviewed and added to the project scope with versioning

### Real-Time Updates

All project data updates in real-time via WebSocket connections. When a GCS team member updates a task status, progress percentage, or adds a comment, the client sees the change reflected immediately without page refresh. Key real-time events include:

- Task status changes (animated transition on Kanban board)
- Progress bar increments
- New messages in project chat
- File uploads and deliverable submissions
- Milestone completion celebrations (confetti animation)
- Team member activity presence indicators

## 11. Authentication & Security

### Authentication Flow

1. Client receives invitation email from GCS admin with secure registration link
2. Registration: email, password (12+ chars, complexity requirements), company name, full name
3. Email verification via one-time link
4. Optional two-factor authentication (TOTP via authenticator app)
5. Login with email/password + optional 2FA
6. Session management: JWT tokens with refresh rotation, 30-min access token, 7-day refresh

### Security Measures

- HTTPS everywhere with HSTS headers
- CSRF protection on all forms
- Rate limiting on authentication endpoints (5 attempts per 15 minutes)
- Input sanitization and XSS prevention
- SQL injection prevention via parameterized queries (Prisma)
- Content Security Policy (CSP) headers
- Role-based access control (RBAC): Admin, Staff, Client Admin, Client User
- Audit logging for all sensitive operations
- File upload validation (type, size, malware scan)
- GDPR-compliant data handling and right-to-delete support

## 14. Dark Mode / Light Mode Implementation

Theme switching is implemented at the CSS variable level with three modes: Light (default), Dark, and System (auto-detect). The theme preference is persisted in a cookie to prevent flash-of-wrong-theme on page load.

### Implementation Strategy

1. CSS variables defined in `:root` (light) and `[data-theme='dark']` (dark) selectors
2. Theme toggle component in the portal top bar and public site navigation
3. System preference detection via `prefers-color-scheme` media query
4. Cookie-based persistence with server-side rendering support
5. Tailwind CSS `dark:` variant classes for component-level overrides
6. Smooth 200ms transition on theme switch for all color properties
7. Image assets provide both light and dark variants where needed (logos, illustrations)

Every component in the design system is built theme-aware from the ground up — no hardcoded colors anywhere. All color references use CSS variable tokens, ensuring a single `data-theme` attribute change cascades the entire visual theme instantly.
